# Me1ody AI Search 复试讲解稿

## 一句话介绍

Me1ody AI Search 是一个面向可信回答的 AI 搜索系统。它不是直接把问题丢给大模型，而是先通过 Tavily 召回网页来源，再进行清洗、质量评分、本地向量检索、重排序，最后让 DeepSeek 基于来源生成带引用的回答。

## 核心流程

1. 用户输入问题。
2. Tavily 进行搜索召回。
3. 后端对搜索结果去重、清洗、识别来源类型并打质量分。
4. 将来源内容切分为 chunk。
5. 使用 embedding 构建本地向量索引。
6. 根据用户问题检索最相关片段，并用 reranker 进行重排序。
7. 将高质量片段作为上下文交给 DeepSeek 生成答案。
8. 对答案中的引用编号进行审计，检查是否引用了真实存在的来源。
9. 前端展示检索链路、阶段耗时、来源可信度、AI 回答和追问建议。

## 可以重点讲的技术点

- RAG：用外部检索结果增强大模型回答，减少幻觉。
- 向量检索：把文本片段编码成向量，用相似度找相关内容。
- 重排序：先粗召回，再精排，提升最终上下文质量。
- 引用审计：回答必须能追溯到来源，避免模型随意编造引用。
- 流式输出：后端用 SSE 逐步推送阶段状态和 token，前端实时渲染。
- 工程化：Docker Compose 编排前后端，提供健康检查和离线 Demo 模式。

## 项目难点与解决

### 搜索不到或结果质量低

普通搜索结果会有重复、广告页、低质量内容。项目通过去重、来源类型识别和质量评分，把更可靠的来源排在前面。

### 大模型可能编造答案

项目要求 DeepSeek 基于检索来源回答，并在生成后进行引用审计。如果没有可靠来源，系统会明确提示不能给出确定结论。

### 现场演示容易受网络影响

第七阶段加入 `DEMO_MODE=true`。该模式使用本地预置来源和回答，不依赖外部 API，也不会消耗额度，但仍保留完整检索链路展示。

### Docker 构建依赖过重

Docker 默认使用 `EMBEDDING_BACKEND=hash`，避免构建阶段下载 PyTorch 和 HuggingFace 模型。本地开发仍可使用 `sentence_transformers` 获得更好的语义检索效果。

## 老师可能问的问题

### 为什么不用大模型直接回答？

直接回答容易出现幻觉，也缺少来源。我的系统通过 RAG 把搜索结果作为上下文，让回答可以追溯到具体来源。

### RAG 的关键步骤是什么？

先召回文档，再切分文本，做向量化，检索相关片段，重排序，最后把片段放进 prompt 让模型回答。

### 如何判断来源可靠？

项目会综合搜索分数、内容长度、域名特征和来源类型，给来源打质量分，并在前端展示可信度。

### 如果搜不到怎么办？

如果没有足够可靠来源，系统不会强行生成确定结论，而是提示用户换问法或补充关键词。

### 项目后续怎么优化？

可以加入更系统的评测集、缓存热门问题、异步任务队列、用户反馈机制，以及更强的来源可信度模型。
